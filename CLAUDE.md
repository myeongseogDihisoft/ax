# AX 프로젝트 개발 원칙

이 문서는 저장소에 체크인된 **팀 공용 규약**입니다. 모든 협업자(사람·AI)는 clone 즉시 아래 원칙을 적용합니다.

---

## 핵심 원칙: TDD (Test-Driven Development)

이 프로젝트의 **모든 코드는 TDD로 작성**합니다. TDD는 단순히 테스트 코드를 만들기 위한 도구가 아니라, **시스템이 가질 수 있는 edge case를 설계 단계에서 미리 드러내는 개발 철학**입니다.

### 사이클: RED → GREEN → REFACTOR

| 단계 | 해야 할 것 | 하지 말아야 할 것 |
|---|---|---|
| **RED** | 실패하는 테스트부터 작성. **happy path뿐 아니라 edge case**를 명시적으로 설계해 테스트로 박아둠 | 구현 먼저 쓰기, happy path 1개로 끝내기 |
| **GREEN** | 테스트를 통과시키는 **최소한의 구현**만 작성 | 테스트가 요구하지 않는 기능·추상화 추가 |
| **REFACTOR** | 테스트가 모두 초록인 상태에서 구조 개선·중복 제거 | 테스트 빨간 상태에서 리팩토링 |

### RED 단계에서 반드시 고려해야 하는 edge case 카테고리

- **입력 경계**: 빈 값, `null`, 최소/최대 길이, 공백, 유니코드, 형식 위반
- **데이터 불변식**: 중복(unique 제약), 외래키 무결성, 소프트 삭제 상태
- **권한/인증**: 비로그인, 타 사용자, 만료된 세션, 권한 상승 시도
- **동시성**: 동시 수정, 레이스 컨디션, 멱등성
- **실패/복원**: DB 예외, 외부 API 타임아웃, 부분 실패
- **성능 한계**: 페이지네이션 경계, N+1, 대량 데이터

### 적용 방법

- **신규 기능**: RED 테스트 목록(시나리오+edge case)을 먼저 제안·합의 → GREEN → REFACTOR
- **버그 수정**: 버그를 재현하는 **RED 테스트부터 추가** → 수정으로 GREEN 전환
- **리팩토링**: 기존 테스트가 부족하면 **특성화 테스트(characterization test)** 로 RED→GREEN 사이클 먼저 확보한 뒤 구조 변경
- **커밋 분리 권장**: `test: ...` (RED) → `feat: ...` (GREEN) → `refactor: ...` 의 3단계 히스토리로 남기면 리뷰어가 의도를 따라가기 쉬움

### 체크리스트 (PR 제출 전)

- [ ] 모든 프로덕션 코드에 대응하는 테스트가 있는가
- [ ] RED 단계에서 설계한 edge case가 테스트로 박혀 있는가
- [ ] 테스트를 지우거나 `@Disabled` 처리하지 않았는가
- [ ] `./gradlew test` 전체 초록인가

## 작업 단위 규약 (Work Granularity)

TDD 사이클이 실제로 돌려면 각 작업 단위가 **한 호흡에 리뷰 가능한 크기**여야 합니다.

### 이슈 한 건 = 파일 수정 **10개 이하**
- 10개 초과가 예상되면 **하위 이슈로 쪼개는 것을 먼저 제안**한다.
- 쪼개기 기준: 도메인별 / 레이어별(Entity vs Service vs Controller) / RED 테스트 그룹별.
- "어쩔 수 없이 10개를 넘겨야 한다"는 판단은 이슈에 이유를 명시하고 리뷰어에게 사전 공유.

### 파일 한 개 = **300줄 미만**
- 300줄에 근접하면 책임 분리 신호로 본다. 클래스/모듈 단위로 쪼갠다.
- 테스트 파일도 동일하게 적용한다. 한 테스트 클래스가 커지면 시나리오별로 분리.
- 생성 파일이 300줄을 넘기게 될 것으로 보이면 **작업 시작 전에 구조 분할을 먼저 제안**한다.

### 예외
- Gradle Wrapper(`gradle-wrapper.jar` 등) 같은 바이너리/기계 생성 파일은 라인 규칙에서 제외.
- 자동 생성되는 스냅샷/마이그레이션 SQL도 예외(단, 명시 표기).

## 기타 규약

- 커밋 메시지 타입: `feat`, `fix`, `refactor`, `test`, `docs`, `chore`
- 브랜치 전략: 작은 단위 PR 지향. `feat/*`, `fix/*`, `refactor/*`, `chore/*`
- 스펙 변경 시 **README.md 먼저 업데이트** → 그 위에서 RED 테스트 작성
